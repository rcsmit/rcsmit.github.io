Identity and Role Model #identity

A User is a single identity across the platform (one email, one login, one global profile). Roles are event-scoped memberships, not user attributes.

model EventMembership {
  id        String  @id @default(cuid())
  userId    String
  eventId   String
  tenantId  String  // denormalized for query efficiency
  roles     Role[]  // array — one record per user per event

  @@unique([userId, eventId])
  @@index([tenantId])
  @@index([eventId])
}

enum Role { ORGANIZER STAFF TEACHER PARTICIPANT }

The same user can hold multiple roles at one event and memberships at multiple events across multiple tenants.

JWT payload contains only: { sub: userId, tenantId, iat, exp }. No role field. Roles resolved per-request per-event from the database.

SiteOwner is a separate model (PlatformUser) with a separate auth flow. SiteOwners are never EventMembership members.

SaaS Tenancy #tenancy

  • Default URL: {slug}.acrofest.app
  • Custom domain (Pro+): tenant sets CNAME, platform issues cert via Caddy + Let's Encrypt. Domain verification: tenant adds a TXT record. Apex and subdomain both supported.
  • Public website rendering: tenant-aware runtime rendering with aggressive caching — not per-tenant SSG builds. Cache invalidated on organizer publish action. Pre-render only homepage and /programme as hot paths.
  • Database: one PostgreSQL database, all records scoped by tenantId. Upgrade path to per-tenant schemas is a data migration, not a redesign.

TypeScript & Code Conventions #conventions

  • strict: true in all tsconfig.json files
  • No any. Use unknown and narrow with Zod.
  • Zod schema is the single source of truth for every API input shape
  • All monetary values: integers in minor units (cents). Never floats.
  • All timestamps: UTC in database. Display converted to event's IANA timezone.
  • All findMany queries: cursor-based pagination. No offset pagination anywhere.
  • No console.log in committed code. Use Pino: logger.info(), logger.warn(), logger.error().
  • All API routes: requestId (UUID v4) generated at entry, attached to Pino logger context, returned in X-Request-Id response header.

API Contract Standards #api-contracts

URL convention: /api/v1/{resource} for public API. /api/internal/{resource} for platform-only endpoints.

Error envelope — canonical shape (no variations):

{
  "error": {
    "code": "SNAKE_CASE_STRING",
    "message": "Human-readable string safe to display to end users.",
    "details": {},
    "requestId": "01HQ7ZXVKJ3M4N5P6Q7R8S9T0U"
  }
}

Pagination envelope:

{
  "items": [],
  "pageInfo": {
    "nextCursor": "clx789abc",
    "hasNextPage": true
  }
}

HTTP status codes — allowed set only:

CodeWhen
200Successful read or update
201Resource created (body contains created resource)
202Async job accepted (body is async envelope)
204Successful delete — empty body
400Malformed JSON, webhook signature invalid
401Missing or expired auth token
402Plan limit reached (makes plan limits semantically distinct from 403)
403Authenticated but insufficient permission
404Resource not found or not visible to caller
409Capacity exceeded, uniqueness conflict, state machine conflict
422Valid JSON but fails Zod validation
429Rate limit exceeded
500Unhandled error — always captured by Sentry
503Readiness check failing, circuit breaker open

Circuit breaker (payment providers): Trip on 5 consecutive failures OR error rate > 50% over 30s. Open state duration: 60s. Redis-shared state across all API instances. Returns 503 with PAYMENT_PROVIDER_UNAVAILABLE when open.

Website cache key format: website:{tenantId}:{path}:{locale}. TTL: 5 minutes. Invalidation uses Redis SCAN (never KEYS/wildcard DEL).

Idempotency: All state-changing endpoints that touch money or send communications accept an Idempotency-Key header (UUID). Keys stored in Redis for 24 hours.

C2 — Money Rules #c2

Invariant Integer minor units

All prices stored as integers in minor units (cents for EUR). Money fields suffixed Cents. Currency stored on every money record. Never floats.

Invariant VAT at purchase time

VAT rate stored immutably at purchase time on the OrderItem (vatRatePct). Never recalculated from current config. V1: one VAT rate per event. Net = grossCents / (1 + vatRatePct), round half-even.

Decision Invoice numbering

Format: {YEAR}-{TENANT_SHORT}-{SEQUENCE}. Sequence per tenant per year with row-level locking. Gap-tolerant: failed invoice numbers are never reused. Must be reviewed by tenant's accountant before regulated B2B use.

Policy Late webhook handling

If payment arrives after reservation expiry and capacity is gone: mark payment PAID, order REQUIRES_RESOLUTION, do NOT fulfill. Queue email to participant. Queue alert to organizer. Pre-create RefundRequest in PENDING_ORGANIZER state. Audit event: order.late_webhook_no_capacity.

C4 — Accommodation Edge Cases #c4

  • Overbooking: Not allowed. Capacity is a hard ceiling. Attempts return 409 with UNIT_AT_CAPACITY.
  • Roommate requests: Free-text field. Organizer resolves manually. No automated matching in v1.
  • Gender/preference: Free-text — no structured enum to avoid assumptions.
  • Waitlist ordering: FIFO by createdAt. No priority tiers in v1.

C6 — Auth Hard Rules #c6

Invariant Refresh token rotation
  • Issued as bcrypt(token, 10) hash — never plaintext
  • Tokens belong to a family (UUID from first login)
  • If used token re-presented → entire family revoked → forced re-login
  • Max 5 active families per user (5 concurrent devices)
Invariant Magic links
  • Max 1 active per user. 15-min expiry. Single-use.
  • crypto.timingSafeEqual for comparison
  • Rate limit: 3 requests per email per hour (Redis sliding window)
  • Token: 32 random bytes, base64url encoded (256 bits of entropy)
Policy TOTP 2FA
  • Required: all ORGANIZER-role users, all PlatformUsers (SiteOwners)
  • Optional: TEACHER, PARTICIPANT
  • Backup codes: 8 single-use codes, stored as bcrypt hashes
  • Reset: chain of authority from ORGANIZER → SiteOwner → second SiteOwner
  • Abuse prevention: max 3 TOTP resets per user per 30 days
Policy Impersonation
  • Requires re-authentication via /api/internal/auth/reauth
  • Impersonation token: max 60 minutes, non-renewable
  • Cannot: impersonate further, modify payment/billing data, delete data, export data
  • Every API call writes to AuditLog with impersonated: true

C9 — Schema Conventions #c9

  • All tables: id String @id @default(cuid()), createdAt DateTime @default(now()), updatedAt DateTime @updatedAt
  • Tenant-scoped: @@index([tenantId]) and @@index([tenantId, id])
  • Event-scoped: @@index([tenantId, eventId])
  • Soft-deletable: deletedAt DateTime? + @@index([deletedAt])
  • Slug uniqueness: event slugs unique per tenant. Profile slugs unique globally.
  • Money fields: suffix Cents. Always Int.
  • Boolean defaults: always explicit (@default(false) or @default(true)), never implicit.
D

Output Protocol

Exactly what to produce, when, and at what granularity

Protocol Rules #protocol-rules

Rule 1 Contracts Before Code

For each phase, produce in this order: (1) Assumptions, (2) File tree, (3) Prisma schema changes, (4) Permission matrix entries, (5) API contracts, (6) Job contracts, (7) Test plan. Only then: implementation code.

Rule 2 One Phase Per Session

Build exactly one phase per session. Do not start the next phase until the current is complete and confirmed.

Rule 3 File Output Format

Each file output as a complete, self-contained code block with filepath as comment on line 1. No truncation. No "// rest of implementation here". If a file is long, output it fully.

Rule 4 No Invented Decisions

If a situation arises that Layer C does not cover, state: OPEN DECISION: {description} and wait for input before proceeding.

Rule 5 No Absolute Language

Do not claim outputs are "production-ready", "complete", or "deployable" without qualification. Instead state: "This phase implements X. Known gaps: Y. Requires review of: Z."
