This guide explains how to work on index.html in Visual Studio Code, push your changes to GitHub, and send them to the map owner via a Pull Request — so she can accept them with one click.

The map owner has her own repository. You work on a fork (your own copy) and propose changes back to her. She reviews and merges. Her live site updates automatically.

  • You need: VS Code with the GitHub extension, a GitHub account, and Git installed
  • This guide assumes you already have those — if not, see the setup notes at the end
How the workflow looks
Owner's repo (github.com/owner/acromap)
↓ you fork it once
Your fork (github.com/yourname/acromap)
↓ you clone to your machine
VS Code (local copy on your computer)
↓ you edit, commit, push
Your fork on GitHub
↓ you open a Pull Request
Owner reviews & merges → live site updates
1
One-time setup

You only do this once. After that you go straight to Section 2 each time you make changes.

1.1 Fork the owner's repository

Forking creates your own copy on GitHub that you can freely edit.

  1. Go to the owner's GitHub repository — for example:
    https://github.com/ownername/acromap
  2. Click the Fork button at the top right
  3. Leave all settings as-is and click Create fork
  4. You now have a copy at https://github.com/yourname/acromap
ℹ️ A fork is not a branch. It is a completely separate repository on your own GitHub account. Changes you make here do not affect the owner's repo until she accepts a Pull Request.
1.2 Clone your fork to VS Code

Cloning downloads the repository to your computer so you can edit it locally.

  1. Open VS Code
  2. Press Ctrl+Shift+P (or Cmd+Shift+P on Mac) to open the Command Palette
  3. Type Git: Clone and press Enter
  4. Paste your fork URL: https://github.com/yourname/acromap
  5. Choose a folder on your computer where the project should live — for example Documents/acromap
  6. Click Open when VS Code asks if you want to open the cloned repo
  7. You will see the project files in the left panel, including index.html
Alternative: You can also clone via the GitHub extension. Click the GitHub icon in the left sidebar → Clone Repository → paste your fork URL.
1.3 Set up the upstream remote

This connects your local copy to the owner's repository so you can pull their changes later (Section 4). You only do this once.

Open the VS Code terminal: Ctrl+` (backtick) or go to Terminal → New Terminal. Then run:

VS Code Terminal
$ git remote add upstream https://github.com/ownername/acromap.git
$ git remote -v
origin https://github.com/yourname/acromap.git (fetch)
origin https://github.com/yourname/acromap.git (push)
upstream https://github.com/ownername/acromap.git (fetch)
upstream https://github.com/ownername/acromap.git (push)

Replace ownername with the owner's actual GitHub username. The second command just confirms both remotes exist.

2
Making changes

This is your regular working cycle each time you update the map.

2.1 Edit the file
  1. In the VS Code file explorer (left panel), click index.html to open it
  2. Make your changes — bug fixes, new features, design updates, etc.
  3. Save the file: Ctrl+S (or Cmd+S on Mac)
⚠️ Important: do not change the Worker URL.
The owner has her own Worker URL in the file pointing to her private Google Sheet. If your development version has a different URL, always revert that specific line before committing. Her line looks like:
fetch('https://acromap-worker.HER-NAME.workers.dev/events')
You can change everything else — just leave that URL alone.
2.2 Commit your changes in VS Code

A commit is a saved snapshot of your changes with a description of what you did.

  1. Click the Source Control icon in the left sidebar (the branching icon, or press Ctrl+Shift+G)
  2. You will see your changed files listed under "Changes"
  3. Hover over index.html and click the + icon to stage it
    Staging means: "include this file in the commit"
  4. In the Message box at the top, type a short description of what you changed. Be specific — for example:
    ✅Fix continent dropdown not updating on event type change
    ✅Add no-results message to calendar view
    ✅Improve mobile layout for filter bar
    ❌updatetoo vague
    ❌changesuseless
  5. Click the ✓ Commit button (or press Ctrl+Enter)
Tip: commit often. One logical change per commit is much easier to review and easier to undo if something goes wrong. Don't bundle ten unrelated fixes into one commit.
2.3 Push to your fork on GitHub

Pushing uploads your local commits to GitHub so they are visible online.

  1. Still in the Source Control panel, click the ⋯ menu (three dots) at the top right
  2. Click Push
  3. Or use the keyboard shortcut: open the Command Palette (Ctrl+Shift+P), type Git: Push and press Enter
  4. VS Code may ask you to sign in to GitHub the first time — follow the prompts
ℹ️ Push vs. Sync: "Push" sends your commits to GitHub. "Sync" does both a pull (download any new changes) and a push. Either works here — use Push if you just want to upload.
3
Sending a Pull Request

A Pull Request (PR) is how you ask the owner to accept your changes into her repository. She will receive an email notification and can merge it with one click.

3.1 Open the Pull Request on GitHub
  1. Go to your fork on GitHub: https://github.com/yourname/acromap
  2. GitHub will usually show a yellow bar saying "This branch is ahead of ownername:main" — click Contribute → Open pull request
  3. If the bar is not showing, click the Pull requests tab → New pull request
  4. Make sure the settings at the top say:
    Base repositoryownername/acromap — branch: main
    Head repositoryyourname/acromap — branch: main
  5. You will see a diff showing exactly what you changed (green = added, red = removed). Review it to make sure everything looks right.
  6. Click Create pull request
3.2 Write a clear description

A good PR description helps the owner understand what changed and why — even if she is not technical.

FieldWhat to write
Title Short summary — e.g. Fix dropdown filters + add no-results message
Description Explain in plain language:
  • What was the problem or what is new
  • What exactly you changed
  • Anything she needs to do after merging (usually: nothing)

Example description:

## What changed
- Continent and country dropdowns now update correctly when an event
  type pill is toggled (previously showed countries with no events)
- Calendar view now shows a "no events found" message instead of
  an empty grid when no events match the current filters

## How to test
After merging, go to the map, select "Retreat" from the event type
pills, and check that only countries with Retreats appear in the
country dropdown.

## Worker URL
I have not changed the Worker URL — it still points to the
production endpoint.
  1. Click Create pull request to submit
  2. The owner receives an email notification from GitHub immediately
3.3 What happens next
  1. The owner reviews the PR on GitHub and can leave comments or questions
  2. If she requests changes, make them in VS Code, commit, and push — the PR updates automatically
  3. When she is happy, she clicks Merge pull request — the changes go live within 1–2 minutes
  4. GitHub will send you a notification when it is merged
The PR stays open until she merges or closes it. You can keep adding commits to the same PR — they appear automatically.
4
Staying in sync with the owner's repo

The owner may make small edits directly on GitHub — for example updating the Worker URL or editing event data. Before you start working on a new feature, always pull the latest version from her repository.

4.1 Pull owner's latest changes into your fork

Open the VS Code terminal (Ctrl+`) and run these two commands:

VS Code Terminal
$ git fetch upstream
From https://github.com/ownername/acromap
a1b2c3d..e4f5g6h main -> upstream/main
$ git merge upstream/main
Updating a1b2c3d..e4f5g6h
Fast-forward
index.html | 12 +++---

Then push the update to your fork so GitHub is also up to date:

VS Code Terminal
$ git push origin main
Do this before you start any new work. If you forget and your fork gets out of date, you may get a merge conflict when you open the next PR.
What to do if you get a merge conflict

A merge conflict happens when both you and the owner edited the same line. VS Code shows conflicts clearly:

  1. VS Code highlights conflicting lines in index.html with <<<<<<<, =======, and >>>>>>> markers
  2. Click Accept Current Change, Accept Incoming Change, or Accept Both above each conflict — choose whichever is correct
  3. Save the file, then stage and commit: Resolve merge conflict
  4. Push to your fork
Most conflicts are easy to resolve because the file is a single index.html — conflicts usually only happen if both of you edited the same JavaScript function.
5
The Worker URL rule

This is the single most important rule when working on this project.

⚠️ Never commit the owner's Worker URL.

The owner has a secret Worker URL that connects to her private Google Sheet:
fetch('https://acromap-worker.HER-NAME.workers.dev/events')

If you are developing locally with your own test Worker, you will have a different URL in your local file. Always revert this line to her URL before committing and pushing.

The cleanest way to handle this is to keep a comment in the file:

// PRODUCTION URL — do not change this line
const res = await fetch('https://acromap-worker.HER-NAME.workers.dev/events');

For your own local testing, you can temporarily swap the URL but always revert before committing. Use VS Code's Source Control diff view to double-check before every commit — the URL line will be highlighted in red/green if it changed.

Alternative: use a config constant. You could define the URL as a constant at the top of the CONFIG block (WORKER_URL: '...'), which makes it easy to see and easy to check before committing.
6
Workflow cheatsheet
Every time you make changes
  • Pull latest from upstream first (git fetch upstream && git merge upstream/main)
  • Edit index.html in VS Code
  • Check the Worker URL is still the owner's URL (not yours)
  • Stage changes in Source Control panel
  • Write a meaningful commit message — what changed and why
  • Commit (Ctrl+Enter)
  • Push to your fork (Source Control → ⋯ → Push)
  • Open a Pull Request on GitHub with a plain-language description
  • Wait for the owner to merge — she gets an email notification
